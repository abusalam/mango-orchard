# Deploying Aamar Malda

Routine deploy steps and the versioning workflow that stamps every build with a `v{semver} · {short-commit}` tag in the footer.

The footer tag is generated by [`App\Support\Version`](../app/Support/Version.php) — read at request time via the view composer in `AppServiceProvider::boot()` and rendered in [`layouts/site.blade.php`](../resources/views/layouts/site.blade.php), [`welcome.blade.php`](../resources/views/welcome.blade.php), and [`components/admin-layout.blade.php`](../resources/views/components/admin-layout.blade.php). It's also surfaced on `/admin/system` (TODO if not added there).

---

## TL;DR — every deploy

```bash
# On the deploy host, in the app directory:
git fetch --tags
git checkout v1.2.3                            # or the release branch tip

# Stamp the commit hash into env so the footer tag includes it
echo "APP_COMMIT=$(git rev-parse HEAD)" >> .env
# Optionally pin the semver too (otherwise it falls through to config/app.php's default)
echo "APP_VERSION=1.2.3" >> .env

# Production-only install + rebuild
composer install --no-dev --optimize-autoloader
npm ci && npm run build

# Apply migrations + (production-safe) seeders
php artisan migrate --force
php artisan db:seed --force --class=Database\\Seeders\\MpcpSeeder      # idempotent — re-reads markdown
# Add any other infra seeders you ship (RolePermissionSeeder, EmailTemplateSeeder, GallerySeeder…)

# Bust caches + reprime
php artisan config:cache
php artisan route:cache
php artisan view:cache
php artisan event:cache

# Restart queue + scheduler workers so they pick up new code
sudo systemctl restart aamar-malda-queue.service     # or `sail restart queue` for Sail-on-prod
sudo systemctl restart aamar-malda-scheduler.service
```

Then verify the footer tag in your browser — it should read `v1.2.3 · <short SHA you just checked out>`.

## How the version tag resolves

`App\Support\Version::tag()` returns `"v{semver} · {short-commit}"`. The two halves resolve independently:

### Semver — `v1.2.3`

Read from `config('app.version')`, which defaults to `env('APP_VERSION', '0.1.0')`.

- **Bump it on every real release** before tagging git: edit `.env` (or the deploy environment's variable store) to set `APP_VERSION=1.2.3`.
- If you don't set the env, the fallback in `config/app.php` (`'0.1.0'`) ships — fine for staging, but you'll want a real value in production so support tickets quote a meaningful number.

### Short commit — `· abc1234`

Resolves through this fallback chain, first hit wins:

1. **`config('app.commit')` → `APP_COMMIT` env var.** This is the **recommended source** in production. Set it at deploy time:
   ```bash
   echo "APP_COMMIT=$(git rev-parse HEAD)" >> .env
   ```
   The full SHA is fine — `Version::shortCommit()` truncates to 7 chars at render time. Storing the full SHA also makes it easier to grep through deploy logs and correlate with GitHub commits.

2. **`.git/HEAD`** — read directly from disk when `APP_COMMIT` is unset. Useful for local development where the `.git` directory is present. Handles all three cases:
   - Symbolic ref (`ref: refs/heads/main`) → follows to `.git/refs/heads/main`
   - Packed refs (`.git/packed-refs`) → scans for the matching ref
   - Detached HEAD (HEAD itself contains a SHA) → uses it directly

3. **Literal `"dev"`** — last resort when neither is available.

> ⚠️ **Don't ship `.git/` to production.** It bloats the deploy and leaks history. Use `APP_COMMIT` instead — it's a single env var and the deploy script writes it automatically.

### Caching

`Version::tag()` caches its result in a static property. Within a single request the file read happens at most once. Across requests, Laravel's config cache (`config:cache` step above) keeps the values memoised until the next deploy.

## Versioning conventions

We follow **semantic versioning** (semver): `MAJOR.MINOR.PATCH`.

| Bump | Trigger |
|---|---|
| **MAJOR** | Breaking change visible to users / API consumers: routes removed, permissions renamed, DB migrations that drop columns, content model restructured (e.g. an MPCP section schema overhauled). |
| **MINOR** | New feature, fully backward compatible: a new module (Gallery, MPCP), a new admin page, a new public route, a new permission. |
| **PATCH** | Bugfix or content tweak: copy edits, CSS polish, validation fixes, a re-seed of MPCP markdown, hotfixes. |

### Git tags

Tag every released version on the default branch:

```bash
git tag -a v1.2.3 -m "Release 1.2.3 — gallery + variety images"
git push origin v1.2.3
```

The tag is what your deploy host's `git checkout v1.2.3` resolves to.

## CI/CD example (GitHub Actions)

```yaml
- name: Stamp deploy with commit hash
  run: |
    echo "APP_COMMIT=${{ github.sha }}" >> $GITHUB_ENV

- name: Build + deploy
  run: |
    composer install --no-dev --optimize-autoloader
    npm ci && npm run build
    # …rsync/upload, then on the remote host:
    #   php artisan migrate --force
    #   php artisan config:cache
    #   etc.
```

The `${{ github.sha }}` expansion gives you the full commit SHA. Pipe it into the remote `.env` (or your secrets manager) and you're done — the footer will display the right hash on the next request.

## Production-only `--no-dev` and faker

`composer install --no-dev` excludes everything in `composer.json`'s `require-dev` block. **One package was moved to `require` so production seeders don't break**: `fakerphp/faker`. If you see this in production logs:

```
Call to undefined function Database\Factories\fake()
```

…it means `--no-dev` stripped faker from a deploy that's running a factory-using seeder. The fix already shipped (faker is in `require`), but if you ever revert that change, gate the demo seeders in `DatabaseSeeder` behind `app()->environment() !== 'production'` instead.

## First-time / fresh-environment deploy

Beyond the routine flow above, a fresh environment needs:

```bash
# 1. .env setup
cp .env.example .env
# Edit .env: APP_KEY, DB credentials, MAIL credentials, APP_VERSION, APP_COMMIT, etc.
# IMPORTANT — branding keys are placeholders in config defaults ("[District Name]").
# Set real values or the placeholders render verbatim in the public chrome:
#   APP_NAME, APP_TAGLINE, APP_OWNER, APP_DISTRICT, APP_DISCLAIMER_OWNER
php artisan key:generate                       # if APP_KEY is blank

# 2. Storage symlink (gallery + variety + listing images live under storage/app/public)
php artisan storage:link

# 3. Migrate + seed everything
php artisan migrate --force
php artisan db:seed --force                    # runs the full DatabaseSeeder chain

# 4. Build assets
npm ci && npm run build

# 5. Cache
php artisan config:cache
php artisan route:cache
php artisan view:cache

# 6. Start daemons
# - queue:work (true daemon, not queue:listen — see hard rule #16 in CLAUDE.md)
# - schedule:work
```

After a fresh deploy, sign in as `admin@malda.gov.in` (superuser, password from `DatabaseSeeder`) and rotate the password immediately.

## Verification

After a deploy, three quick checks:

1. **Footer tag** — open any page and scroll to the footer. The bottom-right under the NIC credits row should read `v{your-semver} · {short SHA}`. If it reads `v0.1.0 · dev`, the env vars weren't picked up; check `php artisan config:show app.version` and `app.commit`.

2. **Status page** — `/admin/system` shows the queue worker heartbeat (`Worker running` pill) and the schedule. If either looks off, restart the corresponding service.

3. **Storage link** — `curl -I https://your-host/storage/test` should 404, not redirect or 500. If it's missing, run `php artisan storage:link`.

## Rolling back

```bash
git checkout v1.2.2                            # the previous tag
echo "APP_COMMIT=$(git rev-parse HEAD)" > /tmp/commit && \
  sed -i "s/^APP_COMMIT=.*/$(cat /tmp/commit)/" .env
echo "APP_VERSION=1.2.2" >> .env

composer install --no-dev --optimize-autoloader
npm ci && npm run build

# Down-migrations are tricky — for most rollbacks just leave the DB schema
# alone (forward compat is on you when adding columns; drop migrations only
# in MAJOR releases where you've explicitly written a safe down path).

php artisan config:cache
sudo systemctl restart aamar-malda-queue.service
sudo systemctl restart aamar-malda-scheduler.service
```

The footer should now read the old version. If a migration that *can't* be reversed shipped between the two tags, you'll need a backup restore — see [`docs/README.md`](README.md#5-database-backup--restore-via-docker) for the pg_dump / pg_restore flow.
